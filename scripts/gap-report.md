# Gap Report — Agent Testing

Issues found during end-to-end agent testing with the Arcadia MCP server.

## Gaps

| # | Scenario | Step | Tool | Issue | Severity | Status |
|---|----------|------|------|-------|----------|--------|
| 2 | Browse LP strategies | Understand APY breakdown | get_strategies | Full strategy list returns `strategy_apy` as a single number. No breakdown into fee APY vs reward APY vs IL. Featured strategies return a different schema with `apy` field. Inconsistent naming. | medium | open (backend) |
| 3 | Browse LP strategies | Compare leverage profitability | get_strategies + get_lending_pools | No tool combines strategy APY with borrow cost to show net leveraged yield. Had to manually cross-reference pool borrow rates with strategy max_leverage. A `get_recommendation`-style tool for "best leveraged strategies" would help. | medium | open (backend) |
| 4 | Browse LP strategies | Identify leverage pool | get_strategies | `max_leverage` map uses raw token addresses as keys (e.g. `0x833589...`). No human-readable pool name. Agent must hardcode address-to-name mapping. | medium | open (backend) |
| 6 | View account positions | Understand liquidation price | get_account_info (account) | `liquidation_price` returned `1.02e-9` (essentially $0). This is technically correct for a PDN strategy but confusing — no context explaining why. A note like "position is delta-neutral, liquidation unlikely" or expressing it as a % drop would be more useful. | medium | open (backend) |
| 7 | View account positions | Interpret asset values | get_account_info (account) | USDbC shows `value: 0` and `value_usd: 0` despite having 201113 units (0.20 USDbC). Unclear if this is an oracle issue or if USDbC is intentionally zero-valued. No explanation in response. | medium | open (backend) |
| 14 | Create account + deposit | Verify new account values | get_account_info (account) | V4 spot account shows `collateral_factor: 0` for USDC and `value_usd` in raw wei-like format (`999999570289523500` for 1 USDC) instead of human-readable. Numeraire is `USD` with address `0x0000...0000` rather than USDC. Confusing for interpreting account state. | medium | open (backend) |
| 16 | Open LP position | Error diagnostics | build_add_liquidity_tx | The 500 error message `"Error fetching web3 data, {}"` gives zero diagnostic info. Root cause was "ERC20: transfer amount exceeds balance" (wallet had insufficient USDC for the deposit). The backend should propagate the Tenderly revert reason to the agent so it can self-correct (e.g. "wallet USDC balance 3.0 < requested 4.0"). | critical | open (backend) |
| 17 | Open LP position | Identify strategy protocol | get_strategies (compact) | Compact strategy list only returns `strategy_id`, `strategy_apy`, and `underlyings` (addresses only). No protocol name, pool name, or tick spacing. Agent cannot distinguish Aerodrome from Uniswap without fetching each strategy individually or using `featured_only=true` (which only shows curated picks). Need `protocol` field in compact list. | high | **closed** — compact list now includes `name`, `protocol`, and underlying `symbol` |
| 18 | Open LP position | Interpret simulation failure | build_add_liquidity_tx | Response returned `tenderly_sim_status: false` but no error message in the tool output. Agent had to visit the `tenderly_sim_url` via WebFetch to discover the revert reason ("ERC20: transfer amount exceeds balance"). The revert reason should be included directly in the tool response when simulation fails. | high | **closed** — tool now detects `tenderly_sim_status: "false"` and returns an explicit warning with the sim URL and common causes |
| 19 | Open LP position | Verify LP was minted | build_add_liquidity_tx | Called `build_add_liquidity_tx` with strategy_id=4 (Aerodrome WETH/USDC), ~$3.99 deposit, no leverage. TX simulated successfully (`tenderly_sim_status: true`) and was confirmed on-chain, but **no LP position was created**. The backend silently swapped USDC→WETH instead of minting a concentrated liquidity position. | critical | **closed** — root cause was MCP tool sending `buy[0].asset_address` = underlying token instead of strategy LP contract. Fixed: now sends `strategy.asset_address` + `strategy.asset_decimals`. Also added minimum_margin guard. |
| 20 | Open LP position | Pre-check minimum margin | get_strategies (detail) | Strategy detail response does not include `minimum_margin` or `risk_factors` despite the `build_add_liquidity_tx` tool description referencing "details.risk_factors". Agent cannot pre-validate whether a deposit amount is sufficient before calling add-liquidity. | high | **closed** — API does return `details.risk_factors` in strategy list. Tool now reads and validates `minimum_margin` before calling the API, rejecting with a clear error if below threshold. |
| 21 | Open LP position | Identify protocol from strategy detail | get_strategies (detail) | `get_strategies(strategy_id=X)` detail response does not include `protocol`, `name`, or position manager address. Only returns pool_address, ticks, ranges, and yield. Agent cannot determine if a strategy is Aerodrome vs Uniswap V3 without calling `featured_only=true` (which only shows curated picks) or trial-and-error with `build_add_liquidity_tx`. Initially misidentified strategy 4 (UniV3) as Aerodrome. Had to use featured list to find strategy 15 was the actual Aerodrome pool. | high | open (backend) |
| 22 | Open LP position | Fetch strategy details | get_strategies (detail) | Strategies 25, 34, and 43 return `500 Internal Server Error` when fetching details individually. No indication these strategies are inactive or unsupported — they appear in the compact list with valid APYs. Agent cannot inspect them to determine protocol. | medium | open (backend) |
| 23 | Open LP position | Compact list missing protocol | get_strategies (compact) | Despite gap #17 being marked as closed with "compact list now includes `name`, `protocol`, and underlying `symbol`", the actual compact response still only returns `strategy_id`, `strategy_apy`, and `underlyings` (addresses only, no symbols). The fix may not be deployed or may require a different parameter. | high | open (backend) |
| 24 | Create account | V4 account can't borrow | build_create_account_tx + build_borrow_tx | Created V4 account with `account_version: 0` (latest) and WETH creditor. Account was created successfully and creditor showed in tx logs, but `build_borrow_tx` reverted with `AccountUnhealthy()`. The tool description doesn't mention that V4 accounts cannot borrow — only mentions V4 spot accounts can't mint LP. Had to create a V3 account to borrow. The `build_create_account_tx` tool should warn that V4 margin accounts have borrowing limitations, or the `account_version` description should clarify which versions support which operations. | high | open (tool description) |
| 25 | Move LP position | NFT approval via build_approve_tx | build_approve_tx + build_deposit_tx | `build_approve_tx` generates an ERC20-style `approve(spender, amount)` call regardless of token type. For ERC721 tokens (LP NFTs), this doesn't work — need `setApprovalForAll(operator, true)` instead. Agent had to manually encode and submit the ERC721 approval via `cast`. The tool should detect ERC721 tokens and generate the correct approval call, or a separate `build_approve_nft_tx` tool should exist. | high | open (tool) |
| 26 | Move LP position | Deposit into V4 account | build_deposit_tx | V4 deposit (function selector `0x95bd34b8`) reverted with `TRANSFER_FROM_FAILED` despite `setApprovalForAll` being granted. V3 deposit (selector `0x656e8729`) succeeded for the same NFT with the same approval pattern. Unclear if V4 deposit requires a different approval mechanism or has a bug. No error context from the revert to diagnose. | medium | open (needs investigation) |
| 27 | Borrow | Account too small to borrow | build_borrow_tx | Account with only an LP NFT (~$3.93 value) was at health_factor=1 with zero debt because collateral_value (0.00166 WETH after haircut) < used_margin (0.002 WETH minimum). Borrow reverted with `AccountUnhealthy()` — no custom error message explaining WHY it's unhealthy. Agent had to decode the error selector manually via `cast 4byte`. The `build_borrow_tx` tool should pre-check health and return a clear error like "collateral_value (X) < minimum_margin (Y), add more collateral". | high | open (tool) |
| 28 | Create account | Indexer delay for new accounts | get_account_info (wallet) | New V4 account didn't appear in `get_account_info(wallet_address)` even after 15+ seconds. Had to parse tx receipt logs manually with `cast` to find the account address. The `build_create_account_tx` response says to call `get_account_info` but should also return the deterministic account address directly (it's derivable from the salt). | medium | open (tool) |
